## 1. 文档目标

本文件定义 RepoRead 的产品定位、范围边界、目标用户、阶段目标与实施原则。

RepoRead 本轮不再把自己定义成“重 RAG、重多 Agent 的研究平台”，而是收敛为：

- 一个本地优先、质量优先的代码阅读与技术书写工作台。
- 一个围绕同一套主控 Agent 运转的仓库理解系统。
- 一个以结构化页面、页面感知问答和研究式写作为核心能力的长期阅读环境。

本文件回答“为什么做”“做什么”和“明确不做什么”。实现细节、数据结构、Agent Loop、Tool 契约、测试方案分别在 [design.md](./design.md) 与 [agent-architecture.md](./agent-architecture.md) 中定义。

## 2. 产品背景

### 2.1 收敛方向

Zread 已验证三件事：

1. 本地扫描仓库是可信起点。
2. 先做目录规划、再生成页面是有效主链路。
3. 版本化落盘和本地浏览能够形成稳定闭环。

RepoRead 本轮不是沿着“更多 Agent、更多检索层、更多平台能力”继续外扩，而是选择收敛主链路。产品判断是：

1. 用户真正持续使用的，不是通用研究平台，而是一个能稳定读代码、写技术内容、反复追问的工作台。
2. 质量瓶颈不在于堆更多并行角色，而在于主控 Agent 是否稳定、页面证据是否充分、审稿是否独立。
3. 问答与研究能力应建立在页面和源码之上，而不是另起一套重型知识底座。

### 2.2 设计来源

本轮设计可概括为：逆向 Zread 的主链路，借鉴成熟 coding agent 的交互和约束，把产品重新收敛成一个本地优先的阅读与写作工作台。

| 来源 | 借鉴重点 |
| --- | --- |
| Zread | 本地扫描、Catalog/Page 两阶段生成、版本化阅读闭环 |
| Claude Code | 工具契约、权限分层、模式切换、上下文压缩、长任务交互方式 |
| Codex | 本地优先、清晰主循环、严格工具边界、plan/work 心智分层 |
| OpenCode | Web/CLI 双端一致性、provider-agnostic 运行方式、面向终端的任务反馈 |
| oh-my-openagent | 对 Agent 能力边界的组织方式，但不复制其多 Agent 平台形态 |
| DeepWiki-Open | 页面化知识组织、研究入口和 Wiki 阅读体验，但不复制其通用 RAG 路线 |

### 2.3 设计结论

RepoRead 的产品结论是：

> 围绕同一套主控 Agent，在 `catalog`、`page`、`ask`、`research` 四种 mode 下工作，输出可验证页面、可追问答案与可沉淀技术书写结果。

## 3. 产品定位

### 3.1 产品定义

RepoRead 是一个面向本地仓库、自托管环境和团队知识沉淀场景的代码阅读与技术书写工作台。系统围绕同一套主控 Agent 运转，并在四种 mode 下切换：

1. `catalog`
   负责本地扫描、目录规划、阅读路径设计和页面范围划分。
2. `page`
   负责按目录严格串行生成页面，完成取证、成文、审稿、发布。
3. `ask`
   负责围绕当前页面、当前版本和当前仓库做页面感知问答，结合实时本地检索补足证据。
4. `research`
   负责围绕复杂问题做多步分析、材料收敛和技术写作，输出比问答更完整的研究结果。

该定义强调三点：

- 同一套主控 Agent，而不是不同任务各自维护一套割裂流程。
- 轻量清单 + 实时本地检索，而不是预先构建重型知识库。
- 阅读与写作同源，所有结论都尽量回落到页面、源码和版本上下文。

其中，“轻量清单”指用于记录导航结构、页面顺序、版本状态与任务恢复线索的轻量文件型清单，不是重型索引库。

### 3.2 产品边界

RepoRead 聚焦“读代码、写技术内容、研究实现细节”，明确做以下事情：

- 做本地扫描、目录规划和结构化页面生成。
- 做结构化页面生产与技术审稿，保证内容可读、可追溯。
- 做页面感知问答、版本浏览和源码对照阅读。
- 做 Web / CLI 双端阅读与操作闭环。

RepoRead 明确不做以下事情：

- 不做通用 RAG 平台。
- 不做 Agent Swarm 控制台。
- 不做用户级 prompt 编排台，也不向用户暴露 prompt 覆盖。
- 不做自动改代码执行器，不承担写代码、跑修改、提交流程的主职责。

### 3.3 核心价值

1. 把“一次性总结”升级为“可导航、可引用、可追问、可继续写作”的持续阅读系统。
2. 让生成、问答、研究共享同一套页面语义和仓库上下文，不拆成两套产品。
3. 以质量优先于吞吐为原则，默认追求稳定、完整、可验证的页面输出。
4. 用最少配置换取可用性，让用户以较低门槛获得稳定、可预期的阅读体验。
5. 让 CLI 与 Web 提供一致的阅读闭环和任务反馈体验，而不是两个互相漂移的产品面。

## 4. 目标用户

| 用户角色 | 核心诉求 | 典型场景 |
| --- | --- | --- |
| 新加入项目的开发者 | 快速建立仓库整体认知，并能沿页面与源码持续追问 | 上手遗留项目、加入新团队、接手陌生模块 |
| 技术负责人 / 架构师 | 沉淀可复用的技术页面与研究结果，降低口头传承成本 | 架构梳理、设计复盘、技术书写 |
| 开源维护者 | 提供比 README 更系统的阅读入口，帮助贡献者理解结构与实现 | 大型仓库导览、贡献路径说明、版本浏览 |
| 平台 / 效率 / 研究团队 | 面向复杂仓库做本地优先的分析、对照与研究，而不是搭建通用知识平台 | 预研、竞品分析、内部知识沉淀 |

这些用户的共同点是：需要高可信阅读和技术书写能力，而不是需要一个能自动改代码的自治 Agent 系统。

## 5. 产品目标与非目标

### 5.1 产品目标

### G1. 建立可信、可持续阅读的仓库知识入口

- 从本地扫描开始，先形成目录规划，再产出结构化页面。
- 页面要可引用、可回溯、可版本化浏览，而不是一次性摘要。

### G2. 让目录规划、页面生成、问答和研究形成统一产品心智

- 用户能在 `catalog`、`page`、`ask`、`research` 四种 mode 下完成连续工作。
- 用户面对的是一套稳定心智模型，而不是多套松散能力拼装。

### G3. 交付高质量、可验证的页面内容

- 页面内容要在结构、证据和可读性上保持稳定水准。
- 生成结果应支持长期阅读、引用和后续扩写。

### G4. 提供页面感知问答与研究能力

- 问答要能围绕当前页面、当前版本和源码上下文持续追问。
- 研究模式围绕复杂问题做多步分析，并输出可沉淀的技术写作结果。

### G5. 以低门槛交付稳定体验

- 用户只需要完成少量必要配置，就能进入阅读、问答和研究主流程。
- 产品默认体验应稳定、清晰、可恢复。

### G6. 完成双端一致的阅读闭环

- 支持版本浏览、源码对照、Web / CLI 双端阅读。
- 支持从生成、阅读、追问到继续研究的连续体验。

### 5.2 非目标

本阶段明确不做以下扩展方向：

1. 不建设基于 SQLite、embedding、向量检索的通用知识底座。
2. 不提供 Agent Swarm 式多 Agent 调度台或用户可见的复杂编排能力。
3. 不开放用户级 prompt 编排、prompt 覆盖和可视化调参台。
4. 不扩展到自动改代码、自动执行命令、自动提交 PR 的执行型代理。
5. 不建设云端 SaaS 账号体系、多人协同编辑和组织级权限系统。
6. 不承诺替代人工评审，也不把系统输出宣称为绝对正确的代码真相。

## 6. 目标形态、分期与实施原则

### 6.1 完整目标形态

完整产品形态应形成以下闭环：

1. 本地仓库接入与最小配置。
2. 轻量清单构建与实时本地检索准备。
3. `catalog` mode 输出目录规划、页面范围和阅读路径。
4. `page` mode 逐页生成、审稿、发布结构化内容。
5. `ask` mode 围绕页面、版本和源码上下文持续追问。
6. `research` mode 围绕复杂问题沉淀更完整的技术写作结果。
7. Web / CLI 双端完成阅读、版本浏览和任务追踪。

### 6.2 实施分期

| 阶段 | 范围 | 目标 |
| --- | --- | --- |
| V1 | 单主循环、四种 mode、严格串行页面生成、独立审稿、页面感知问答、Web / CLI 阅读闭环 | 先把代码阅读与技术书写工作台做扎实 |
| V1.5 | 研究模式增强、版本对比、再生成体验优化、质量反馈闭环 | 让复杂问题分析和长期阅读更顺手 |
| V2 | 在不破坏主链路的前提下扩展可选能力 | 保持本地优先与质量优先，不回到平台堆叠路线 |

### 6.3 实施原则

1. 严格串行
   `catalog` 完成后再进入 `page`；页面逐页生成，不以跨页并发换取表面吞吐。

2. 单页内并行取证
   允许在单页内部并行收集源码、符号、引用和上下文证据，但页面成文与发布仍由单一主流程收束。

3. 审稿独立会话
   `fresh.reviewer` 必须与页面生成会话分离，避免作者视角污染审稿判断。

4. 用户配置最少
   用户只提供必要仓库信息、模型凭证和少量偏好；系统不要求用户理解复杂编排细节。

5. 系统负责模型适配、降级与系统内建 prompt 调优
   模型选择、能力探测、失败降级、默认系统内建 prompt 调优都由系统内建处理，不向用户暴露 prompt 覆盖。

6. 轻量清单 + 实时本地检索
   主链路依赖轻量清单和实时本地检索，不引入额外的重型持久化检索层作为前置条件。

## 7. 关键用户旅程

### 7.1 首次使用：初始化与 Provider 配置

**目标**：用户在本地完成最小必要配置，确认系统可读仓库、可调模型、可进入主链路。

**主流程**

1. 用户在仓库目录执行 `repo-read init`。
2. CLI 向导识别当前仓库、工作目录权限、默认语言和可见 Git 状态。
3. 向导要求用户只配置三类角色：
   - `main.author`
   - `fork.worker`
   - `fresh.reviewer`
4. 每个角色只暴露 `model` 与 `fallback_models` 两个字段；凭证按 Provider 维度保存。
5. 系统执行连接测试、工具能力检查、只读权限检查和最小样例调用。
6. 用户可立即执行 `repo-read providers` 与 `repo-read doctor` 复查配置健康状态。

**闭环要求**

- CLI 是首次入口，不是隐藏后台。
- Web 读取同一份项目状态，进入 `/projects` 后必须能看到初始化结果和下一步入口。
- 用户无需理解 prompt、调度器和底层 schema，即可完成可用配置。

### 7.2 首次生成：`catalog -> page -> fresh.reviewer -> validator -> publish`

**目标**：用户第一次就理解 RepoRead 的主链路是“先规划阅读顺序，再逐页成文，再独立审稿，再发布版本”。

**主流程**

1. 用户执行 `repo-read generate` 或在项目页点击“生成新版本”。
2. 系统先做仓库预检查，产出轻量清单，建立阅读顺序与章节依赖。
3. 系统进入 `catalog` mode，输出页面目录、页面边界、依赖关系和建议阅读路径。
4. `catalog` 完成后进入 `page` mode，按目录严格串行生成页面。
5. 每一页内部允许并行取证，但草稿只能由 `main.author` 收束成文。
6. 每页草稿生成后，必须交给独立会话的 `fresh.reviewer` 审稿。
7. 审稿通过或修订完成后，交给确定性 `validator` 做结构、引用、链接和边界校验。
8. 全部页面完成后发布新版本，进入可浏览状态，并保留 job 记录。

**成功标准**

- 用户能在 Web 与 CLI 中看到当前 mode、当前页、当前阶段和最近一次审稿结果。
- 页面失败不会污染已完成页面；支持 `interrupt`、`resume` 和逐页重试。
- 发布后的版本立即可进入书架、版本页和页面页阅读。

### 7.3 浏览与追问：书架、版本、目录树、正文、引用抽屉、Chat Dock

**目标**：Web 和 CLI 一起构成阅读闭环，而不是一个负责生成、另一个只做展示。

**主流程**

1. 用户在 `/projects` 进入书架，选择项目。
2. 在项目页查看版本列表、最近 job、当前版本状态和重新生成入口。
3. 进入版本页后，看到 Catalog 目录树、阅读顺序和章节依赖。
4. 进入页面页后，正文区阅读页面，引用抽屉按需展开源码片段。
5. 用户从页面内的内部链接、推荐阅读或搜索页继续跳转。
6. 用户在页面级 `Chat Dock` 中追问，系统优先读取当前页与引用，再决定是否追加实时本地检索。
7. 需要终端快速确认时，用户可执行 `repo-read ask`，沿用当前项目、版本和页面上下文。

**成功标准**

- 书架、版本、页面、引用和追问构成同一阅读路径。
- 页面切换不会丢失当前版本、当前页和最近引用上下文。
- 回答必须能回链到文件、页面或 commit，而不是只给自然语言结论。

### 7.4 深度研究：复杂问题追根溯源

**目标**：用户围绕复杂问题进行比页面问答更长的检索和写作链路，但仍停留在本地优先、可追溯的边界内。

**主流程**

1. 用户在页面级 `Chat Dock` 或 CLI 中切换到 `research` mode。
2. 系统先给出研究目标、子问题和预期证据来源。
3. 系统围绕当前页、相关页面、Git 历史和源码片段逐步扩展检索范围。
4. 研究过程中的输出明确区分：
   - 事实
   - 推断
   - 待确认项
5. V1 中，研究结果首先作为当前项目/版本上下文中的研究结果或研究笔记存在，主要从页面级 `Chat Dock`、CLI `research` 和 job 页进入；后续版本可再吸收为页面或附录候选。

**成功标准**

- 用户能看到研究不是“更长的闲聊”，而是更长的证据链。
- 研究结果比普通问答更完整，但仍受页面边界、源码证据和版本上下文约束，并先以研究结果或研究笔记形态沉淀。
- 研究中所有关键结论都能回链到页面、文件或 commit。

## 8. 信息架构

### 8.1 CLI 命令结构

CLI 不是“备用入口”，而是初始化、生成、任务追踪、诊断和快速问答的一级界面。V1 命令结构如下：

| 命令 | 定位 | 关键输出 |
| --- | --- | --- |
| `repo-read init` | 首次初始化项目与 Provider 向导 | 角色级模型配置、权限检查、保存结果 |
| `repo-read providers` | 查看与编辑 Provider 健康状态 | Provider、模型、回退链、测试结果 |
| `repo-read generate` | 发起新版本生成 | 当前 mode、当前页、当前阶段、最近 review / validate 结果 |
| `repo-read jobs` | 查看 job 列表与 job 详情 | 任务状态、失败原因、可 `resume` 页、可重试页 |
| `repo-read browse` | 启动本地阅读站点 | 访问地址、默认项目、默认版本 |
| `repo-read ask` | CLI 快速问答 | 当前项目/版本/页面上下文、引用摘要 |
| `repo-read research` | CLI 深度研究 | 研究计划、进展、事实/推断/待确认、研究笔记 |
| `repo-read doctor` | 环境与配置诊断 | Provider 连通性、仓库只读能力、任务恢复性 |
| `repo-read versions` | 浏览版本清单 | 版本状态、发布时间、来源 job |

**命令设计约束**

1. `generate` 必须支持 `interrupt`、`resume` 和失败页重试，不以“整批重跑”代替恢复。
2. `jobs` 与 `versions` 必须能互相跳转，让用户在任务视角和阅读视角之间切换。
3. `ask` 与 `research` 必须继承当前项目和版本上下文，避免让用户重复描述环境。

### 8.2 Web 路由结构

Web 不是 CLI 的附属展示层，而是技术书阅读器、版本浏览器和页面内追问界面。

| 路由 | 页面定位 | 核心内容 |
| --- | --- | --- |
| `/projects` | 书架页 | 项目卡片、最近版本、最近 job、继续阅读入口 |
| `/projects/:project` | 项目页 | 项目概览、版本列表、生成入口、任务摘要 |
| `/projects/:project/versions/:version` | 版本页 | Catalog、章节依赖、阅读顺序、版本信息 |
| `/projects/:project/versions/:version/pages/:slug` | 页面页 | 目录树、正文、引用抽屉、页面级 `Chat Dock` |
| `/projects/:project/versions/:version/search` | 搜索页 | 页面结果、文件结果、跳转入口 |
| `/jobs/:jobId` | job 页 | 当前阶段、当前页、最近审稿状态、失败原因、恢复操作、相关研究结果/研究笔记入口 |

**路由设计约束**

1. `/projects` 是主入口；若保留 `/`，应直接重定向到 `/projects`。
2. 版本页和页面页必须分离，避免把 Catalog 与正文混成单页工作台。
3. 页面页是阅读闭环中心，不能退化成通用 Agent 控制台。
4. V1 不新增独立 research 路由体系；研究结果先作为当前项目/版本上下文中的研究结果或研究笔记存在。

## 9. 详细功能需求

### 9.1 Provider 与模型配置

| 编号 | 需求 | 优先级 | 验收标准 |
| --- | --- | --- | --- |
| FR-001 | 系统支持全局配置与项目级覆盖；项目级优先 | P0 | 同一项目内读取到明确生效配置 |
| FR-002 | 对用户只暴露三种角色：`main.author`、`fork.worker`、`fresh.reviewer` | P0 | UI 与 CLI 不出现额外角色配置入口 |
| FR-003 | 每个角色只允许配置 `model` 与 `fallback_models` | P0 | 配置文件与向导字段一致，无 prompt 覆盖入口 |
| FR-004 | 角色级模型配置需要通过连接测试、工具能力测试和最小样例调用 | P0 | 用户能看到是凭证失败、模型能力不足还是速率受限 |
| FR-005 | 界面与 CLI 中的角色名称统一显示为 `fork.worker`，不再使用并列别名 | P1 | 配置键、界面文案和任务日志中的角色名称一致 |
| FR-006 | 敏感配置必须掩码展示并与项目内容分离保存 | P0 | 不在正文、日志和版本内容中泄露凭证 |

### 9.2 系统内建 prompt 调优

| 编号 | 需求 | 优先级 | 验收标准 |
| --- | --- | --- | --- |
| FR-007 | 系统内建 prompt 调优用于 `catalog`、`page`、`ask`、`research` 四种 mode | P0 | 不同 mode 可加载对应系统策略 |
| FR-008 | 系统内建 prompt 调优由系统维护，可随版本演进，但不向用户开放 prompt 覆盖 | P0 | CLI 与 Web 均无“自定义 prompt”入口 |
| FR-009 | 用户可选择质量预设，但预设只影响系统内部的系统内建 prompt 调优和模型策略，不暴露原文 | P1 | 用户能理解效果差异，但看不到内部 prompt 文本 |

### 9.3 仓库理解与轻量清单

| 编号 | 需求 | 优先级 | 验收标准 |
| --- | --- | --- | --- |
| FR-010 | 生成前必须做仓库预检查并输出 Repo Profile | P0 | 可见语言、规模、入口、模块边界和 Git 基线 |
| FR-011 | 系统只生成轻量清单，用于记录目录结构、阅读顺序、页面状态、版本状态和恢复线索；主链路不得引入向量库、SQLite 或重型符号数据库 | P0 | 主链路在无向量库、无 SQLite、无重型符号数据库前提下可运行 |
| FR-012 | 清单与实时本地检索共同工作：清单负责导航与恢复，检索负责按需取证 | P0 | `ask` 与 `research` 可在无预构建重索引下完成回答 |
| FR-013 | 轻量清单至少记录层级、顺序、依赖、覆盖文件和状态等恢复所需信息 | P0 | Catalog 展示与 job 恢复都能读取这些信息 |

### 9.4 Catalog 与页面生成

| 编号 | 需求 | 优先级 | 验收标准 |
| --- | --- | --- | --- |
| FR-014 | `catalog` mode 必须先产出阅读顺序、章节依赖和页面边界，再允许进入页面生成 | P0 | 无 Catalog 时不得直接批量生成页面 |
| FR-015 | `page` mode 在目录确认后必须严格串行；不得跨页并发生成正文 | P0 | job 页可见当前页唯一推进 |
| FR-016 | 单页内部允许并行取证，由 `fork.worker` 收集片段和候选证据，再由 `main.author` 统一成文 | P0 | 页面草稿来源清晰，主稿责任单一 |
| FR-017 | 每页草稿必须经过独立会话的 `fresh.reviewer` 审稿，再经过确定性校验器 | P0 | 未审稿或未校验页面不得发布 |
| FR-018 | 页面状态至少区分待生成、生成中、审稿中、修订中、失败、已发布等用户可见状态 | P0 | CLI 与 job 页展示一致且可理解 |
| FR-019 | 长任务必须支持暂停后继续推进与失败页重试，不覆盖已通过页面 | P0 | 用户可从最近未完成位置继续推进 |

### 9.5 代码问答与深度研究

| 编号 | 需求 | 优先级 | 验收标准 |
| --- | --- | --- | --- |
| FR-020 | `ask` mode 先读当前页、最近引用和相关页面，再决定是否做实时本地检索 | P0 | 文档足够时不重复扫仓库 |
| FR-021 | 追加检索时，优先使用本地只读检索链路，例如 `grep`、`find`、`read`、`git` | P0 | 回答中可说明引用来自文件、页面或 commit |
| FR-022 | 代码问答结果必须可追溯到文件、页面或 commit，且引用可回跳 | P0 | Web 可打开引用抽屉，CLI 可打印引用摘要 |
| FR-023 | `research` mode 允许更长检索链路，但输出必须区分事实、推断、待确认 | P0 | 回答结构明确，不能把推断写成事实 |
| FR-024 | V1 中研究结果应先沉淀为当前项目/版本上下文中的研究结果或研究笔记，而不是一次性聊天记录；后续版本可再吸收为页面或附录候选 | P1 | 用户可从 `Chat Dock`、CLI `research` 和 job 页进入这些结果，并看到后续采用情况 |

| 编号 | 需求 | 优先级 | 验收标准 |
| --- | --- | --- | --- |
| FR-025 | CLI 负责初始化、生成、任务、诊断和快速问答 | P0 | 核心链路不依赖浏览器也可完成 |
| FR-026 | Web 负责书架、目录树、正文、引用、页面内追问和版本切换 | P0 | 长阅读与对照体验在 Web 中完整 |
| FR-027 | Web 与 CLI 必须共用同一任务状态、版本状态和引用语义 | P0 | 同一 job 在两端展示一致 |
| FR-028 | `browse` 必须能把 CLI 用户带入当前项目和版本，而不是打开空白首页 | P0 | 启动后可直达指定阅读上下文 |
| FR-029 | `jobs`、job 页、版本页、页面页之间必须可互相跳转 | P1 | 用户能从失败页回到阅读页，也能从阅读页回到生成上下文 |

## 10. 界面与交互要求

### 10.1 设计原则

1. Web 体感应接近技术书阅读器，而不是通用 Agent 控制台。
2. 阅读优先，任务信息可见但不能压过正文。
3. Catalog、页面、引用、追问应形成连续阅读体验，而不是多个割裂面板。
4. CLI 输出必须像可靠的构建与审稿工具，而不是只会刷原始日志。

### 10.2 核心页面定义

| 页面 | 核心区域 | 关键交互 | 必须覆盖的状态 |
| --- | --- | --- | --- |
| 书架页 `/projects` | 项目卡片、最近版本、最近 job、继续阅读入口 | 打开项目、继续阅读、进入 job | 空书架、初始化未完成、有进行中 job |
| 项目页 `/projects/:project` | 项目概览、版本列表、生成入口、诊断摘要 | 新建版本、查看版本、查看 job | 无版本、单版本、多版本、存在失败 job |
| 版本页 `/projects/:project/versions/:version` | Catalog 目录树、章节依赖、阅读顺序、版本元信息 | 跳到首读页、切换版本、打开搜索 | Catalog 缺失、版本发布中、版本可读 |
| 页面页 `/projects/:project/versions/:version/pages/:slug` | 目录树、正文、引用抽屉、页面级 `Chat Dock` | 点引用、切页、内部跳转、页面内追问 | 页面加载中、页面不存在、引用抽屉展开、Chat 展开 |
| 搜索页 `/projects/:project/versions/:version/search` | 搜索框、结果列表、结果类型过滤 | 搜页面、搜文件、跳结果 | 无结果、搜索中、结果过多 |
| job 页 `/jobs/:jobId` | 阶段轨迹、当前页、最近审稿结果、失败原因、恢复动作、相关研究结果/研究笔记入口 | 暂停后继续推进、重试失败页、跳到版本或页面、查看研究结果 | 运行中、中断、待恢复、部分失败、完成 |

### 10.3 Wiki 页面交互细节

**目录树**

1. 默认展示清晰的章节层级。
2. 必须可见页面顺序、章节依赖和推荐阅读方向。
3. 当前页、前一页、后一页需要有明显阅读提示。

**正文区**

1. 页面头部固定显示标题、版本、commit、页面边界和最近生成信息。
2. 正文以可持续阅读为目标，避免把任务日志混入主体内容。
3. 页面内引用、内部链接和推荐阅读应形成稳定跳转。
4. 页面不得越界，不应把其他页面的核心结论偷渡进来。

**引用抽屉**

1. 点击引用后展开，不改变正文滚动位置。
2. 至少展示文件路径、行号、代码片段、所属 commit 或版本信息。
3. 用户可从引用抽屉回跳正文，也可继续打开源码定位。

**Chat Dock**

1. 默认最小化。
2. 打开后显示当前项目、版本、页面和最近引用上下文。
3. 回答需标明引用来自页面、文件还是 commit。
4. 用户可以从页面级问答切入深度研究，并查看当前版本上下文中的研究结果或研究笔记；页面阅读区始终保持主位。

### 10.4 CLI 交互要求

1. 所有高频命令输出应结构化、可扫描，并持续显示当前 mode。
2. `generate` 必须持续显示当前页、当前阶段、最近 review 结果、最近 validate 结果。
3. `jobs` 和 `doctor` 必须把失败原因说清楚，而不是让用户自行翻日志。
4. `ask` 与 `research` 默认继承当前项目和当前版本上下文。
5. 所有失败输出都应给出下一步动作，例如继续推进、重试、回到版本页或重新诊断。

## 11. 内容质量标准

### 11.1 Catalog 标准

1. Catalog 必须体现阅读顺序，而不是只给目录快照。
2. Catalog 必须显式表达章节依赖，让用户知道先读什么、后读什么。
3. 每个 Topic 都必须映射到真实文件集合和明确页面边界。
4. Catalog 的颗粒度应支持“先上手，再深挖”，但不能把整个仓库压缩成几页空泛综述。

### 11.2 页面标准

1. 页面开头必须说明本页范围、为何值得读和与前后页的关系。
2. 页面正文必须围绕本页边界展开，不得越界吞并其他页面的核心结论。
3. 核心结论必须有证据来源；没有证据的内容要删除或降级为推断。
4. 页面引用必须可定位、可打开、可验证，并通过结构与链接校验。
5. 架构图、流程图只服务于解释，不用于装饰和凑版面。

### 11.3 审稿与研究标准

1. `fresh.reviewer` 的审稿结论必须包含 `blockers`、`factual_risks`、`missing_evidence`、`scope_violations`。
2. 页面若存在 blocker，不得进入发布态；必须修订后重审。
3. 问答先给结论，再给证据；证据不足时必须直说不足。
4. 深度研究必须把事实、推断、待确认项分开书写，并以可继续引用的研究结果或研究笔记沉淀，不能混写。
5. 超出仓库边界、版本边界或证据边界时，系统必须显式说明。

## 12. 非功能需求与质量优先指标

### 12.1 质量优先原则

1. 默认采用质量优先策略；若提供预设，默认应为质量预设。
2. 目录合理性、页面完整性、引用正确性优先于速度和吞吐。
3. 不承诺固定生成时长；系统优先交付可验证内容，而不是表面更快的批量结果。
4. 深度研究可以更慢，但必须可解释、可中断、可恢复。

### 12.2 可恢复性与可见性原则

| 原则 | 系统要求 |
| --- | --- |
| 长任务可中断 | 生成与研究都必须支持 `interrupt` |
| 长任务可恢复 | 中断、崩溃或刷新后都可 `resume` 到最近未完成阶段 |
| 长任务可重试 | 支持单页重试、失败页重排和重新审稿 |
| 过程可见 | 用户始终能看到当前页、当前阶段、最近审稿状态和失败原因 |
| 错误可见 | 错误不得只藏在日志文件中，必须在 CLI 与 Web 中直观呈现 |
| 状态可回溯 | job、版本、页面三层状态都能追溯到来源与变更 |

### 12.3 质量指标

| 指标 | 目标 |
| --- | --- |
| Catalog 结构校验 | 发布前必须通过 |
| 页面结构与链接校验 | 发布前必须通过 |
| 页面引用可定位性 | 发布前必须通过 |
| 审稿结论字段完整性 | 发布前必须通过 |
| 问答可追溯引用能力 | 样例仓库抽检时逐项通过 |
| 人工抽检 | 样例仓库验收项逐项通过 |

### 12.4 安全与可靠性

1. 默认只读，不上传源码，不把仓库内容作为外部服务前置条件。
2. 单页失败不影响已发布页面与历史版本的可读性。
3. 敏感配置不明文展示，日志默认不落敏感信息。
4. 工具调用和检索必须限定在当前仓库边界内。
5. 所有版本都必须可回溯到来源 job、提交基线和发布时间。

## 13. 验收标准与发布门槛

### 13.1 V1 功能验收

V1 发布前，以下能力必须可用：

1. 支持 `init -> generate -> interrupt -> resume -> browse -> ask` 完整链路。
2. CLI 具备 `init/providers/generate/jobs/browse/ask/research/doctor/versions` 基础能力。
3. Web 具备 `/projects`、项目页、版本页、页面页、搜索页、job 页。
4. 角色级模型配置仅包含 `main.author`、`fork.worker`、`fresh.reviewer`，且均支持 `fallback_models`。
5. Catalog、页面、引用抽屉、页面级 `Chat Dock` 和版本切换形成完整阅读闭环。
6. 页面级 `fresh.reviewer` 与确定性校验器闭环可工作。

### 13.2 样例仓库验收

至少使用以下三类仓库人工验收：

1. 小型工具类仓库。
2. 中型 Web 应用仓库。
3. 中型后端服务或 Monorepo 仓库。

每个仓库至少检查：

1. Catalog 是否体现阅读顺序与章节依赖。
2. 页面是否存在明显事实错误、越界总结或缺证据结论。
3. 引用是否能跳转到正确文件片段、页面或 commit。
4. 页面内问答是否遵循“先页面、后实时本地检索”的链路。
5. `interrupt`、`resume`、失败页重试和版本切换是否稳定。
6. Provider 切换、回退链和审稿链路是否稳定。

### 13.3 发布阻断项

出现以下任一情况不得发布：

1. 核心 CLI 命令或核心 Web 路由不可用。
2. 发布前未通过 `fresh.reviewer`，或修订后未重新审稿。
3. 页面未通过结构、引用或链接校验。
4. 页面存在 blocker、严重 `factual_risks`、明显 `missing_evidence` 或 `scope_violations` 未处理。
5. 版本生成后无法浏览、无法定位源码引用，或问答无法给出可追溯引用。
6. Provider 配置链路无法完成、错误不可解释，或任务状态无法恢复。
