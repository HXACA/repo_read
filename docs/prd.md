## 1. 文档目标

本文件定义 RepoRead 的产品定位、范围边界、目标用户、阶段目标与实施原则。

RepoRead 本轮不再把自己定义成“重 RAG、重多 Agent 的研究平台”，而是收敛为：

- 一个本地优先、质量优先的代码阅读与技术书写工作台。
- 一个围绕同一套主控 Agent 运转的仓库理解系统。
- 一个以结构化页面、页面感知问答和研究式写作为核心能力的长期阅读环境。

本文件回答“为什么做”“做什么”和“明确不做什么”。实现细节、数据结构、Agent Loop、Tool 契约、测试方案分别在 [design.md](./design.md) 与 [agent-architecture.md](./agent-architecture.md) 中定义。

## 2. 产品背景

### 2.1 收敛方向

Zread 已验证三件事：

1. 本地扫描仓库是可信起点。
2. 先做目录规划、再生成页面是有效主链路。
3. 版本化落盘和本地浏览能够形成稳定闭环。

RepoRead 本轮不是沿着“更多 Agent、更多检索层、更多平台能力”继续外扩，而是选择收敛主链路。产品判断是：

1. 用户真正持续使用的，不是通用研究平台，而是一个能稳定读代码、写技术内容、反复追问的工作台。
2. 质量瓶颈不在于堆更多并行角色，而在于主控 Agent 是否稳定、页面证据是否充分、审稿是否独立。
3. 问答与研究能力应建立在页面和源码之上，而不是另起一套重型知识底座。

### 2.2 设计来源

本轮设计可概括为：逆向 Zread 的主链路，借鉴成熟 coding agent 的交互和约束，把产品重新收敛成一个本地优先的阅读与写作工作台。

| 来源 | 借鉴重点 |
| --- | --- |
| Zread | 本地扫描、Catalog/Page 两阶段生成、版本化阅读闭环 |
| Claude Code | 工具契约、权限分层、模式切换、上下文压缩、长任务交互方式 |
| Codex | 本地优先、清晰主循环、严格工具边界、plan/work 心智分层 |
| OpenCode | Web/CLI 双端一致性、provider-agnostic 运行方式、面向终端的任务反馈 |
| oh-my-openagent | 对 Agent 能力边界的组织方式，但不复制其多 Agent 平台形态 |
| DeepWiki-Open | 页面化知识组织、研究入口和 Wiki 阅读体验，但不复制其通用 RAG 路线 |

### 2.3 设计结论

RepoRead 的产品结论是：

> 围绕同一套主控 Agent，在 `catalog`、`page`、`ask`、`research` 四种 mode 下工作，输出可验证页面、可追问答案与可沉淀技术书写结果。

## 3. 产品定位

### 3.1 产品定义

RepoRead 是一个面向本地仓库、自托管环境和团队知识沉淀场景的代码阅读与技术书写工作台。系统围绕同一套主控 Agent 运转，并在四种 mode 下切换：

1. `catalog`
   负责本地扫描、目录规划、阅读路径设计和页面范围划分。
2. `page`
   负责按目录严格串行生成页面，完成取证、成文、审稿、发布。
3. `ask`
   负责围绕当前页面、当前版本和当前仓库做页面感知问答，结合实时本地检索补足证据。
4. `research`
   负责围绕复杂问题做多步分析、材料收敛和技术写作，输出比问答更完整的研究结果。

该定义强调三点：

- 同一套主控 Agent，而不是不同任务各自维护一套割裂流程。
- 轻量清单 + 实时本地检索，而不是预先构建重型知识库。
- 阅读与写作同源，所有结论都尽量回落到页面、源码和版本上下文。

其中，“轻量清单”指用于记录导航结构、页面顺序、版本状态与任务恢复线索的轻量文件型清单，不是重型索引库。

### 3.2 产品边界

RepoRead 聚焦“读代码、写技术内容、研究实现细节”，明确做以下事情：

- 做本地扫描、目录规划和结构化页面生成。
- 做结构化页面生产与技术审稿，保证内容可读、可追溯。
- 做页面感知问答、版本浏览和源码对照阅读。
- 做 Web / CLI 双端阅读与操作闭环。

RepoRead 明确不做以下事情：

- 不做通用 RAG 平台。
- 不做 Agent Swarm 控制台。
- 不做用户级 prompt 编排台，也不向用户暴露 prompt 覆盖。
- 不做自动改代码执行器，不承担写代码、跑修改、提交流程的主职责。

### 3.3 核心价值

1. 把“一次性总结”升级为“可导航、可引用、可追问、可继续写作”的持续阅读系统。
2. 让生成、问答、研究共享同一套页面语义和仓库上下文，不拆成两套产品。
3. 以质量优先于吞吐为原则，默认追求稳定、完整、可验证的页面输出。
4. 用最少配置换取可用性，让用户以较低门槛获得稳定、可预期的阅读体验。
5. 让 CLI 与 Web 提供一致的阅读闭环和任务反馈体验，而不是两个互相漂移的产品面。

## 4. 目标用户

| 用户角色 | 核心诉求 | 典型场景 |
| --- | --- | --- |
| 新加入项目的开发者 | 快速建立仓库整体认知，并能沿页面与源码持续追问 | 上手遗留项目、加入新团队、接手陌生模块 |
| 技术负责人 / 架构师 | 沉淀可复用的技术页面与研究结果，降低口头传承成本 | 架构梳理、设计复盘、技术书写 |
| 开源维护者 | 提供比 README 更系统的阅读入口，帮助贡献者理解结构与实现 | 大型仓库导览、贡献路径说明、版本浏览 |
| 平台 / 效率 / 研究团队 | 面向复杂仓库做本地优先的分析、对照与研究，而不是搭建通用知识平台 | 预研、竞品分析、内部知识沉淀 |

这些用户的共同点是：需要高可信阅读和技术书写能力，而不是需要一个能自动改代码的自治 Agent 系统。

## 5. 产品目标与非目标

### 5.1 产品目标

### G1. 建立可信、可持续阅读的仓库知识入口

- 从本地扫描开始，先形成目录规划，再产出结构化页面。
- 页面要可引用、可回溯、可版本化浏览，而不是一次性摘要。

### G2. 让目录规划、页面生成、问答和研究形成统一产品心智

- 用户能在 `catalog`、`page`、`ask`、`research` 四种 mode 下完成连续工作。
- 用户面对的是一套稳定心智模型，而不是多套松散能力拼装。

### G3. 交付高质量、可验证的页面内容

- 页面内容要在结构、证据和可读性上保持稳定水准。
- 生成结果应支持长期阅读、引用和后续扩写。

### G4. 提供页面感知问答与研究能力

- 问答要能围绕当前页面、当前版本和源码上下文持续追问。
- 研究模式围绕复杂问题做多步分析，并输出可沉淀的技术写作结果。

### G5. 以低门槛交付稳定体验

- 用户只需要完成少量必要配置，就能进入阅读、问答和研究主流程。
- 产品默认体验应稳定、清晰、可恢复。

### G6. 完成双端一致的阅读闭环

- 支持版本浏览、源码对照、Web / CLI 双端阅读。
- 支持从生成、阅读、追问到继续研究的连续体验。

### 5.2 非目标

本阶段明确不做以下扩展方向：

1. 不建设基于 SQLite、embedding、向量检索的通用知识底座。
2. 不提供 Agent Swarm 式多 Agent 调度台或用户可见的复杂编排能力。
3. 不开放用户级 prompt 编排、prompt 覆盖和可视化调参台。
4. 不扩展到自动改代码、自动执行命令、自动提交 PR 的执行型代理。
5. 不建设云端 SaaS 账号体系、多人协同编辑和组织级权限系统。
6. 不承诺替代人工评审，也不把系统输出宣称为绝对正确的代码真相。

## 6. 目标形态、分期与实施原则

### 6.1 完整目标形态

完整产品形态应形成以下闭环：

1. 本地仓库接入与最小配置。
2. 轻量清单构建与实时本地检索准备。
3. `catalog` mode 输出目录规划、页面范围和阅读路径。
4. `page` mode 逐页生成、审稿、发布结构化内容。
5. `ask` mode 围绕页面、版本和源码上下文持续追问。
6. `research` mode 围绕复杂问题沉淀更完整的技术写作结果。
7. Web / CLI 双端完成阅读、版本浏览和任务追踪。

### 6.2 实施分期

| 阶段 | 范围 | 目标 |
| --- | --- | --- |
| V1 | 单主循环、四种 mode、严格串行页面生成、独立审稿、页面感知问答、Web / CLI 阅读闭环 | 先把代码阅读与技术书写工作台做扎实 |
| V1.5 | 研究模式增强、版本对比、再生成体验优化、质量反馈闭环 | 让复杂问题分析和长期阅读更顺手 |
| V2 | 在不破坏主链路的前提下扩展可选能力 | 保持本地优先与质量优先，不回到平台堆叠路线 |

### 6.3 实施原则

1. 严格串行
   `catalog` 完成后再进入 `page`；页面逐页生成，不以跨页并发换取表面吞吐。

2. 单页内并行取证
   允许在单页内部并行收集源码、符号、引用和上下文证据，但页面成文与发布仍由单一主流程收束。

3. 审稿独立会话
   `fresh reviewer` 必须与页面生成会话分离，避免作者视角污染审稿判断。

4. 用户配置最少
   用户只提供必要仓库信息、模型凭证和少量偏好；系统不要求用户理解复杂编排细节。

5. 系统负责模型适配、降级与 prompt 调优
   模型选择、能力探测、失败降级、默认 prompt 调优都由系统内建处理，不向用户暴露 prompt 覆盖。

6. 轻量清单 + 实时本地检索
   主链路依赖轻量清单和实时本地检索，不引入额外的重型持久化检索层作为前置条件。

## 7. 关键用户旅程

### 7.1 首次使用：Provider 向导 + 首次生成

**目标**：用户在不手改配置文件的前提下完成首次配置并生成第一版 Wiki。

**主流程**

1. 用户在仓库目录执行 `repo-read init` 或 `repo-read config`。
2. 系统进入向导，依次完成：
   - 选择预设：`quality / balanced / budget / local-only`
   - 选择 Provider
   - 录入凭证
   - 分配角色到模型
   - 选择语言、并发度、排除规则、索引策略
   - 连接测试与保存
3. 用户执行 `repo-read generate`。
4. 系统显示仓库画像、预计阶段、关键配置摘要。
5. 生成完成后可进入 `browse` 或 `ask`。

**成功标准**

- 用户无需理解底层配置 schema 也能完成配置。
- 向导能发现不可用 Provider、模型不支持工具调用、Embedding 未配置等问题。
- 生成完成后有明确入口进入阅读或问答。

### 7.2 日常使用：Web 中重新生成已有项目

**目标**：用户能在 Web 中发起新版本生成并追踪完整过程。

**主流程**

1. 用户进入首页或项目页。
2. 点击“重新生成”。
3. 进入生成工作台，调整：
   - 版本备注
   - 模型预设或角色映射
   - 语言与质量模式
   - 排除规则
4. 启动任务。
5. 实时查看阶段、Worker、失败页、重试、当前工具调用摘要。
6. 完成后跳转到新版本。

**成功标准**

- 用户不依赖 CLI 也能完整发起生成。
- 页面刷新后可恢复任务视图。
- 用户能区分“卡在 Catalog”还是“卡在某几个页面”。

### 7.3 浏览 Wiki：从总览深入到源码

**目标**：用户在长时间阅读中保持方向感，并能随时跳到源码片段。

**主流程**

1. 用户打开项目的某个版本页面。
2. 左侧目录树保持导航，主内容区阅读页面，右侧工具区承载 TOC / Source / Chat。
3. 用户点击引用，打开源码抽屉。
4. 用户点击推荐阅读或内部链接继续阅读。
5. 用户在搜索中跳转到其他页面或符号。

**成功标准**

- 阅读区不被任务面板打断。
- 源码抽屉打开后不破坏当前阅读位置。
- 当前页面、版本、commit、生成时间始终可见。

### 7.4 页面内追问：代码问答

**目标**：用户围绕当前页面继续追问，而不是另开一个脱离上下文的聊天窗口。

**主流程**

1. 用户在页面内打开 Chat Dock。
2. 系统自动注入当前页面、当前版本、最近引用和推荐相关页面。
3. 用户提问。
4. 系统先尝试文档答案，再判断是否需要实时读源码。
5. 回答流式输出，并区分引用来源：
   - `wiki`
   - `code`
   - `research`

**成功标准**

- 文档足够回答时不重复做重检索。
- 实时检索时要显式展示来源。
- 用户能一键清空会话。

### 7.5 深度研究：复杂问题追根溯源

**目标**：用户对复杂问题发起多步研究，而不是拿到一次性结论。

**主流程**

1. 用户在 Chat Dock 或独立 Research 入口切换到“深度研究”。
2. 系统生成研究计划，拆出 2-5 个子问题。
3. 系统围绕页面、源码、符号、版本信息做多步证据收集。
4. 必要时调用可选外部文档检索。
5. 系统产出“研究计划 -> 研究进展 -> 结论”结构化回答。

**成功标准**

- 用户能看到系统在研究什么，而不是只看到最终答案。
- 研究模式默认慢但稳，强调证据与边界。
- 回答可以回链到页面和源码。

## 9. 详细功能需求

### 9.1 Provider 与配置中心

| 编号 | 需求 | 优先级 | 验收标准 |
| --- | --- | --- | --- |
| FR-001 | 支持全局配置和项目级配置覆盖 | P0 | 项目级优先于全局配置 |
| FR-002 | 支持 Anthropic、OpenAI、Google、OpenRouter、Ollama、兼容 OpenAI 接口 | P0 | 每类 Provider 至少一套通过验证的样例 |
| FR-003 | 支持按角色配置模型：`router/planner/catalog/page/reviewer/chat/research/embedder` | P0 | 未配置角色可回退到默认模型 |
| FR-004 | 支持预设：`quality`、`balanced`、`budget`、`local-only` | P0 | 新项目首次配置默认 `quality` |
| FR-005 | API Key 与敏感配置以安全方式存储，UI 中只显示掩码值 | P0 | 不明文回显 |
| FR-006 | 配置向导支持连接测试、能力校验、错误解释 | P0 | 用户知道是鉴权失败还是模型不支持 |
| FR-007 | 支持为每个角色配置回退链 | P1 | 主模型失败时可切换到后备模型 |
| FR-008 | 支持导入/导出非敏感配置 | P1 | 导出文件不含明文密钥 |

### 9.2 仓库画像、索引与生成任务

| 编号 | 需求 | 优先级 | 验收标准 |
| --- | --- | --- | --- |
| FR-009 | 生成前必须执行仓库预检查并产出 Repo Profile | P0 | 可见仓库语言、规模、入口、框架线索 |
| FR-010 | 生成前必须构建最小可用索引：文件清单、页面检索索引、代码片段索引 | P0 | Chat 不依赖临时全仓扫描 |
| FR-011 | 生成流程必须拆为 `预检查 -> 索引 -> Catalog -> Page -> Review/Validation -> Publish` | P0 | 事件流可见阶段切换 |
| FR-012 | 页面生成采用任务队列和 Worker Pool | P0 | 页面状态至少区分 `queued/running/reviewing/retrying/failed/done` |

## 10. 界面与交互要求

### 10.1 设计原则

1. 阅读优先，而不是任务面板优先。
2. 任务管理、阅读理解、对话追问三种心智必须分层。
3. Provider 配置不能只是一张表单，而应是“可解释、可验证、可推荐”的向导。
4. 用户不需要理解底层 Agent 术语，也能完成主流程。

### 10.2 核心页面定义

| 页面 | 核心区域 | 关键交互 | 必须覆盖的状态 |
| --- | --- | --- | --- |
| 首页 `/` | 项目列表、最近任务、最近研究会话、快速入口 | 打开项目、发起生成、继续问答 | 空状态、任务失败提醒、有进行中任务 |
| Provider 中心 `/settings/providers` | Provider 卡片、角色路由表、健康状态、连接测试 | 新增密钥、测试模型、切换预设、保存 | 未配置、部分可用、全部可用、校验失败 |
| 生成工作台 `/generate` | 配置向导、任务摘要、阶段时间线、Worker 表、失败页列表 | 启动、取消、恢复、重试失败页 | 未开始、运行中、部分失败、完成、取消 |
| 项目概览 `/projects/:project` | 最新版本、版本列表、最近任务、快捷操作 | 打开版本、重新生成、对比版本 | 无版本、单版本、多版本、有运行任务 |
| Wiki 页面 `/projects/:project/:version/:slug` | 目录、正文、工具区、源码抽屉、Chat Dock | 切页、切版本、点引用、点内部链接、搜索 | 页面加载中、页面不存在、Chat 展开、源码抽屉展开 |
| 搜索页 `/projects/:project/:version/search` | 搜索框、结果列表、过滤器 | 输入关键词、跳结果 | 无结果、结果过多、搜索中 |
| 对比页 `/projects/:project/:version/compare` | 版本选择器、变化列表、变化详情 | 选择基线版本、跳转到具体页面 | 无可比版本、变更过多 |

### 10.3 Wiki 页面交互细节

**目录区**

1. 默认展示完整信息层级。
2. 当前主题和当前分组有持续高亮。
3. 支持输入即过滤。

**正文区**

1. 顶部固定显示页面标题、版本、commit、生成时间。
2. 页面内支持锚点目录。
3. Mermaid 支持全屏。
4. 代码块支持复制。

**工具区**

1. 默认展示 TOC。
2. 点击引用后切到源码抽屉。
3. 打开 Chat 后切到对话模式，但不覆盖正文。

**Chat Dock**

1. 默认最小化。
2. 打开后显示当前上下文徽标。
3. 回答需标明来自 Wiki、代码还是研究。

### 10.4 CLI 交互要求

1. 所有高频命令输出应结构化、可扫描。
2. `generate` 以阶段和 Worker 为主，不以原始日志刷屏为主。
3. `ask` 默认保留当前项目和当前版本上下文。
4. 所有命令在失败时都给出下一步建议。

## 11. 内容质量标准

### 11.1 Catalog 标准

1. 页数建议 10-30 页，复杂仓库允许上探但需保持可读。
2. 每个 Topic 都必须映射到真实文件集合。
3. 目录同时兼顾“快速上手”和“深入理解”。

### 11.2 页面标准

1. 页面开头必须回答“这一页为什么值得读”。
2. 段落要偏工程说明文，不偏营销文。
3. 无证据的结论必须删除或显式降级为推测。
4. 图表用于解释结构和流程，不作装饰。

### 11.3 问答标准

1. 先给结论，再给证据和展开。
2. 文档能答时不做重度检索。
3. 代码问题必须带来源。
4. 超出仓库边界时明确说明边界。

## 12. 非功能需求与质量优先指标

### 12.1 质量优先原则

1. 默认模式为 `quality`。
2. 目录合理性、页面完整性、引用正确性优先于速度。
3. 深度研究默认允许更慢，但必须可解释。

### 12.2 时间指标（建议区间）

| 场景 | 建议区间 | 说明 |
| --- | --- | --- |
| 小型仓库 Catalog | 2-8 分钟 | 依赖仓库复杂度和模型 |
| 单页生成 | 2-6 分钟 | 复杂页面可更长 |
| 小型仓库完整 Wiki | 10-30 分钟 | 默认质量模式 |
| 中型仓库完整 Wiki | 20-60 分钟 | 默认质量模式 |
| Chat 首 token 延迟 | 1-5 秒 | 文档命中快于源码检索 |
| 深度研究首阶段返回 | 3-10 秒 | 先给计划，再进入研究 |

### 12.3 质量指标

| 指标 | 目标 |
| --- | --- |
| Catalog 结构校验通过率 | 100% |
| 页面结构校验通过率 | >= 95% |
| Mermaid 可渲染率 | >= 95% |
| 核心段落引用覆盖率 | >= 90% |
| 代码问答引用命中率 | >= 90% |
| 人工抽检通过率 | >= 90% |

### 12.4 安全与可靠性

1. 任务可恢复。
2. 单页失败不影响整体可见性。
3. 敏感配置不明文展示。
4. 日志默认不落敏感信息。
5. 只读工具必须严格限定仓库边界。

## 13. 验收标准与发布门槛

### 13.1 V1 功能验收

V1 发布前，以下能力必须可用：

1. CLI 初始化、配置、生成、浏览、问答。
2. Web 首页、Provider 中心、生成工作台、项目概览、Wiki 页面。
3. 版本化落盘与版本切换。
4. 目录树、源码抽屉、页面内 Chat。
5. 角色级模型配置与连接测试。
6. 页面级 Reviewer + 确定性校验闭环。

### 13.2 样例仓库验收

至少使用以下三类仓库人工验收：

1. 小型工具类仓库。
2. 中型 Web 应用仓库。
3. 中型后端服务或 Monorepo 仓库。

每个仓库至少检查：

1. 目录是否覆盖核心模块。
2. 页面是否有明显事实错误。
3. 引用是否可跳转到正确片段。
4. 页面内问答是否能正确引用页面或源码。
5. Provider 切换和回退是否稳定。

### 13.3 发布阻断项

出现以下任一情况不得发布：

1. 核心 CLI 命令或核心 Web 路由不可用。
2. 版本生成后无法浏览或无法定位源码引用。
3. 大量页面缺失引用或结构破损。
4. 问答无法引用仓库页面或源码。
5. Provider 配置链路无法完成或错误不可解释。
