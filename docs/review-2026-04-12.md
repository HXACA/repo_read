## 7. 设计优化建议（速度 / 质量 / 渐进式披露）

这一节不是 blocking finding，而是基于当前架构状态给出的下一步优化建议。重点不是“再加更多能力”，而是让系统把高成本质量手段用在真正值得的地方。

### 7.1 把“固定 preset”升级为“按页面复杂度自适应”

当前 `QualityProfile` 基本是静态 preset 映射：

- `packages/core/src/config/quality-profile.ts`
- `packages/core/src/generation/generation-pipeline.ts`

这套设计的优点是简单、可预期；但它的问题也很明显：

- 简单页会被过度处理，浪费时间和 token；
- 难页又可能因为 preset 选得保守而质量不够；
- 同一本书里不同页面的成本结构差异很大，但现在只能全局统一。

更好的方向是做一个轻量的 `PageComplexityScore`，在页面级动态决定：

1. 是否真的需要 `fork.worker`。
2. `forkWorkers` 开几个。
3. reviewer 要不要做严格 citation verify。
4. revise 最多给几轮。

可直接用的输入信号包括：

- `covered_files` 数量
- 涉及目录分散度
- 是否跨语言 / 跨层
- 历史 revise 次数
- 首轮 draft 的 citation 密度
- reviewer 的 blocker 类型

这样质量不会下降，反而更容易把预算集中在难页上。

### 7.2 把生成流水线改成“快路径 -> 触发式升级”的渐进式质量模型

当前设计已经有 quality / balanced / budget preset，但本质上还是“开局就决定整页跑多重”。

我更建议把它变成明确的渐进式质量披露：

1. 第一轮先跑便宜路径：
   - 少量 evidence
   - 小 step budget
   - reviewer 只做结构/覆盖率/明显事实错
2. 只有命中风险信号时才升级：
   - evidence 不足
   - section 无引用
   - 跨页关系断裂
   - reviewer 命中 factual risk / missing evidence
3. 升级后只重跑必要环节，而不是整页 full replay。

这比今天的“preset 直接决定整页成本”更合理，因为：

- 大多数页其实不需要最高强度 reviewer；
- 质量问题通常集中在少数 section，不在整页；
- 用户体感速度会明显更好。

一句话概括：不要默认把每一页都当成 hardest page 处理。

#### 7.2.1 这套机制在 RepoRead 里具体意味着什么

当前实现更像：

`catalog -> 每页按 preset 固定预算执行 -> reviewer -> revise -> validate`

也就是说，一旦项目选了 `quality` 或 `balanced`，每一页基本都按同样的预算模型跑：

- 同样数量级的 `fork.worker`
- 同样的 `drafterMaxSteps`
- 同样的 `reviewerMaxSteps`
- 同样的 citation verify 强度

问题在于，页面复杂度并不均匀：

- “目录/模块概览页”通常很便宜；
- “跨层调用链 / 核心状态机 / 并发模型”这类页面才是真正昂贵页。

所以更合理的目标流程应该是：

`catalog -> page fast path -> risk detection -> selective escalation -> diff review -> validate`

核心变化只有一句话：

- 先用便宜路径证明“这页其实很简单”；
- 只有证明失败时，才升级到更重的质量路径。

#### 7.2.2 一个建议中的页面状态机

我建议把页面级执行显式拆成 4 层：

1. **L0 Fast Path**
   - 单 worker 或无 worker
   - 小 drafter budget
   - reviewer 只做轻审
2. **L1 Evidence Upgrade**
   - 增加 evidence 覆盖
   - 允许重做 outline
3. **L2 Review Upgrade**
   - reviewer 提高 verify 强度
   - 增加 citation spot-check 数
4. **L3 Hard Page**
   - 只给少数真正困难页面
   - 更高 revise budget
   - 更严格 reviewer

重要的是：

- 页面不是一开始就进入 L3；
- 只有触发信号出现时，才从 L0 往上升级；
- 一旦某页在 L0 就通过，不再消耗更高层预算。

#### 7.2.3 Fast Path 默认该做什么

对大多数页面，我建议第一轮默认只做这些事：

1. evidence 只拿最小必要集合
2. `forkWorkers` 从 0 或 1 开始
3. drafter 用保守 step budget
4. reviewer 只判断：
   - 是否偏题
   - 是否明显缺证
   - 是否有明显事实错误
5. validate 继续保持确定性全量执行

也就是说，fast path 不是“降低质量标准”，而是：

- 先用最便宜的手段判断这页是否足够好；
- 把昂贵质量动作延后到有证据表明“它不够好”的时候。

#### 7.2.4 哪些信号会触发升级

这套机制最关键的是触发条件要清晰、确定，不要靠“感觉这页可能难”。

我建议至少用以下信号：

**草稿阶段信号**

- draft 被截断
- metadata 提取出的 citations 过少
- section-level citation density 过低
- outline 某些 section 没有足够 evidence coverage

**review 阶段信号**

- reviewer 命中 `missing_evidence`
- reviewer 命中 `factual_risks`
- reviewer 命中 `scope_violations`
- reviewer issue 集中在少数 section

**静态复杂度信号**

- `covered_files` 太多
- 文件分布跨多个目录/层
- 页面涉及跨语言或跨 runtime 边界
- 首轮就出现多轮 revise

这些信号可以分成两类：

1. **开局前信号**
   - 用于决定是否直接跳过 L0
2. **执行中信号**
   - 用于决定是否从 L0 升到 L1/L2/L3

#### 7.2.5 升级时到底升级什么

“升级”不应该是一个抽象动作，而应该明确映射到运行参数变化。

RepoRead 里最自然的升级维度是：

1. `forkWorkers`
2. `forkWorkerConcurrency`
3. `drafterMaxSteps`
4. `reviewerMaxSteps`
5. `reviewerVerifyMinCitations`
6. `maxRevisionAttempts`

一个简单的升级表可以是：

| Level | `forkWorkers` | `drafterMaxSteps` | `reviewerMaxSteps` | `reviewerVerifyMinCitations` | 用途 |
| --- | --- | --- | --- | --- | --- |
| L0 | 0-1 | 8-12 | 4-6 | 0-1 | 默认快路径 |
| L1 | 2 | 12-18 | 6-8 | 1-2 | 证据不足时 |
| L2 | 2-3 | 18-24 | 10-12 | 2-3 | reviewer 风险较高时 |
| L3 | 3 | 24-30 | 12-15 | 3+ | 真正困难页 |

这里的重点不是数值本身，而是：

- 升级动作要和触发原因对应；
- 不同原因触发的升级，不一定所有参数一起加。

例如：

- `missing_evidence` 优先加 `forkWorkers`
- `factual_risks` 优先加 reviewer verify 强度
- `draft truncated` 优先让 draft 更短或按 section 重写，不一定继续加 budget

#### 7.2.6 升级后不要整页 full replay

这是这个设计能不能真正提速的关键。

如果每次升级都意味着：

- 重新收集整页 evidence
- 重新生成整页 outline
- 重新审整页

那它只是“多了一层状态机”，并没有真正更快。

所以我建议升级后的默认策略是：

1. **Evidence 升级**
   - 只补 reviewer 指出的缺证 section
2. **Outline 升级**
   - 只重排受影响 section 的 cite mapping
3. **Draft 升级**
   - 优先 section-level revise，而不是整页重写
4. **Review 升级**
   - reviewer 只重查受影响 section，再抽样回归

这和下一节提到的“差分重审”是配套设计，不应该分开落。

#### 7.2.7 一个具体例子

以两类页面为例：

**例 1：项目目录总览页**

- `covered_files` 少
- 主题集中
- 首轮草稿 citation 足够
- reviewer 没打出 factual risk

这类页应该：

- L0 起步
- 一轮过
- 不进入升级

**例 2：请求生命周期 / 核心引擎页**

- `covered_files` 多
- 跨 handler / service / model / config
- reviewer 首轮指出 2 个事实风险和 3 个缺证 section

这类页应该：

- 可以直接从 L1 起步，或者 L0 后快速升到 L2
- 补 evidence 时只盯 reviewer 指出的 section
- 第二轮 reviewer 重点复核这几个 section，而不是整页重读

这两种页如果继续用同一固定预算跑，系统一定是在某一边浪费。

#### 7.2.8 这套机制和“渐进式披露”是一回事

这里的 progressive disclosure 不是只指 UI，而是指运行时也遵循同一原则：

1. 先给系统一个最小可判断结果
2. 只有发现风险，才披露更多计算、更多 evidence、更多 review 深度
3. 把重成本动作延迟到“需要它”的时刻

所以它本质上不是“性能优化技巧”，而是一个统一的设计哲学：

- 对用户，先展示 summary，再按需展开 evidence
- 对系统，先跑 fast path，再按需升级质量

前后端如果都遵循这个原则，产品会更统一。

#### 7.2.9 我会怎么落地这件事

如果要实做，我建议分 3 步，而不是一次性重写 pipeline：

**Step 1：先加页面复杂度评分和执行 level 记录**

- 给每页算一个 `complexityScore`
- 给 job/page state 记录本页当前 `executionLevel`
- 先不改太多流程，只把观测打出来

**Step 2：把 evidence/reviewer 升级做成参数化**

- 把 `forkWorkers` / reviewer verify 强度做成 page-level 可调
- 先支持 L0 -> L1 -> L2，不急着做最复杂的 L3

**Step 3：再做差分 revise / 差分 review**

- 这是收益最大的一步，但也最依赖前两步的数据结构
- 没有 section-level issue mapping，就做不好差分升级

这三步里，真正的顺序不要反：

- 先有“能观测页面复杂度和升级原因”的结构
- 再有“能改参数”的机制
- 最后做“只重跑必要部分”的优化

### 7.3 把 reviewer 从“整页重审”收敛成“差分重审”

现在 revise 循环虽然存在，但 reviewer 仍然偏向对整页重新做一轮判断。这里还有提速空间。

建议引入更明确的差分协议：

1. reviewer 输出 issue 时，必须绑定到 section 或 claim cluster。
2. drafter revise 后，记录“哪些 section 被改了、哪些 citation 被替换了”。
3. 下一轮 reviewer 默认只重查受影响 section，再抽样一小部分 untouched section 做回归。

这会同时提升速度和质量：

- 速度：避免整页重复读。
- 质量：issue 能被追踪闭环，不会“改了又忘、忘了又改”。

如果后面要继续做质量硬化，我认为这里的 ROI 高于继续加大 `reviewerMaxSteps`。

### 7.4 把“渐进式披露”从 UI 模式上升为系统级原则

当前 Web 已经有一些渐进式披露雏形：

- `review-button.tsx` 默认只显示状态，点开才看细项
- `citation-popover.tsx` 默认只显示 citation chip，点击才拉 snippet
- `page-reader-client.tsx` 的 TOC / chapter nav 可折叠

这说明方向是对的，但使用还比较局部。更值得做的是把 progressive disclosure 当成整体设计原则，而不是“有些弹窗用了 popover”。

我建议统一成三层：

1. **Summary-first**
   - 先给用户最小可决策信息
2. **Evidence-on-demand**
   - 只有用户要验证时才展开证据
3. **Trace-for-debug**
   - 只在排障/高级用户场景暴露 agent trace / tool trace / review trace

如果这个原则统一下来，很多界面和 API 都会自然变简单。

### 7.5 在阅读器里更系统地使用渐进式披露

这个项目特别适合多用渐进式披露，因为读代码本来就是“先建立轮廓，再向下钻取”的行为。

我觉得最值得加的几个点：

#### A. Version 页不要一次性把全部页面平铺成同权目录

当前 `version-client.tsx` 是整份 reading order 平铺列表。更适合的方式是：

- 默认只展示 section 级目录和每节前 2-3 页；
- 点击 section 再展开页级目录；
- 同时显示这一节的摘要、页数、主题范围。

这能明显降低首次认知负担，尤其是长书。

#### B. Page 页默认展示“正文 + 最少元信息”，其余按需展开

现在页面头部已经有 review status / revision count，但还可以继续做：

- 默认只显示 summary / page position / review badge
- “审阅意见”“引用统计”“相关页面”“页面元数据”作为独立折叠块
- citation 默认展示少量关键条目，剩余条目点开再看

这样页面更像“读书”，不是“把所有系统内部状态直接摊给用户”。

#### C. Ask 默认给短答，证据和推理延迟展开

`ask-stream.ts` 现在已经有 citation 事件和相对紧凑的输出格式。下一步可以更进一步：

- 默认只返回短答 + 1-3 个关键 citations
- 用户点击“展开依据”后再显示更多 evidence
- “思考过程 / tool trace / route classifier 决策”只在 debug 模式出现

这会让问答既更快，也更像面向阅读任务，而不是面向 agent 开发者。

### 7.6 把“质量”从 reviewer 压力，前移成结构化中间产物质量

当前系统已经有 `evidence_ledger`、`outline`、`draft metadata` 这些中间层，这是好事。下一步最值得优化的是：把质量问题尽量前移到这些中间层暴露。

例如：

1. `OutlinePlanner` 输出 section 时，顺手输出每节的 evidence sufficiency score。
2. drafter 产出后，先做确定性的 section-level citation density scan。
3. reviewer 主要处理“解释是否成立”“结构是否合理”，而不是花预算查那种确定性就能发现的缺证。

这能减少 reviewer 被迫承担“质量垃圾回收器”的角色。

一个简单判断标准是：

- 能靠结构化数据算出来的，不要留给 reviewer 靠 prompt 自觉发现。

### 7.7 为速度目标引入“感知速度”设计，而不只是总耗时优化

这个项目还有一个很重要但还没完全系统化的点：用户感受到的速度，未必等于总 token 成本最低。

建议把“感知速度”单独设计：

1. catalog 完成后立刻可见目录草案，而不是等整本书完成。
2. 单页 draft 完成后，可以显式标成 `draft preview` 供预读。
3. review / validate 结果作为后续状态补上。
4. Ask 首屏先回一句短答，扩展证据后到。

这其实也是渐进式披露，只不过披露的是“完成度”而不是“信息量”。

对 RepoRead 这种长任务产品，我认为这类设计的收益很高。

### 7.8 一个我会优先采纳的组合方案

如果只选一组最值得落地的设计优化，我会建议：

1. 页面复杂度评分
2. 快路径 -> 升级路径的渐进式质量执行
3. reviewer 差分重审
4. 阅读器 / Ask 的 summary-first + evidence-on-demand

这四项是同一条主线：

- 后端不要一开始就把所有页面拉满质量成本；
- 前端也不要一开始就把所有细节摊给用户；
- 系统和界面都遵守“先给最小可决策信息，风险出现后再升级”的原则。

如果这样演进，RepoRead 的速度和质量不会互相打架，反而会更容易同时提高。
