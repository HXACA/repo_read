# RepoRead Agent 重设计 Spec

> 日期：2026-04-07
> 状态：已在对话中达成设计共识，现整理为正式书面 spec
> 范围：围绕更轻量、更接近 `claude-code` 的本地代码阅读代理，重写当前 RepoRead 的产品、工程与 Agent 设计文档
> 目标文档：
> - `docs/prd.md`
> - `docs/design.md`
> - `docs/agent-architecture.md`

---

## 1. 目标

RepoRead 不应该继续往“重多 Agent、重 RAG、重研究系统”的方向漂移，而应该收敛成一个高质量、本地优先、以代码阅读和技术书写作为核心能力的系统。

新的总方向是：

- 本地优先
- 质量优先
- 单主控循环
- 轻量文件存储
- 实时本地检索
- 独立审稿
- 用户侧最小化配置

这份 spec 是下一轮重写主文档时的设计依据。

---

## 2. 核心产品判断

RepoRead 不是“一个带很多 Agent 的 AI Wiki 生成器”。

RepoRead 应当是：

- 一个本地仓库阅读工作台
- 一个按严格阅读顺序生成技术书式 Wiki 的系统
- 一个通过更像人类工程师的本地检索方式回答代码问题的系统
- 一个把审稿视为独立第二意见，而不是主控自我润色的系统

它的体感目标更接近 `claude-code` 做深度本地代码阅读，而不是一个传统 RAG 应用。

---

## 3. 主体架构

### 3.1 总体结构

RepoRead 应采用：

`单主循环 + 两种委派原语 + 确定性校验器`

不再保留大量并列的一等 Agent，例如：

- `CatalogAgent`
- `PageWriterAgent`
- `ChatAgent`
- `ResearchSynthesizerAgent`

等这类互相平行的主体。

正确做法是：

- 只有一个主控 Agent 实现负责主线推进
- 这个主控 Agent 运行在不同 mode 下：
  - `catalog`
  - `page`
  - `ask`
  - `research`
- 委派是窄而明确的
- 确定性校验器始终独立于主控 Agent

### 3.2 这样做的原因

这样可以保留：

- 章节连续性
- “作者在写一本书”的强心智
- 更简单的 runtime
- 更少的上下文碎片
- 更低的架构膨胀风险

这也更接近真正好用的 coding agent 的工作方式：一条主推理链，必要时才做局部委派。

---

## 4. 委派语义

RepoRead 只保留两种委派语义。

### 4.1 `fork worker`

用途：

- 单页内部并行取证
- 小节级证据收集
- 局部子总结
- 窄范围、不重叠的检查任务

性质：

- 继承父上下文
- 继承父 system framing
- 只接收 directive，不接收完整 briefing
- 不拥有最终判断权
- 不写最终页面
- 不允许递归委派
- 只返回紧凑的结构化局部结果

典型使用场景：

- 并行检查 3 个候选文件
- 独立总结 3 个小节的证据
- 对单页里的 2 条实现链路做并行比对
- 为若干结论并行收集证据

### 4.2 `fresh reviewer`

用途：

- 独立技术审稿
- 隔离式第二意见
- 对主控初稿做事实挑战
- 在必要时做独立复核

性质：

- 全新会话
- 不继承主控的思维链和运行上下文
- 收到完整 briefing
- 允许重新执行本地检索
- 只输出 review 结果，不接管作者角色

典型使用场景：

- 审核某一页初稿
- 质疑无证据结论
- 查找遗漏、矛盾或证据薄弱点
- 判断页面是否跑偏章节边界

### 4.3 明确非目标

RepoRead 不应变成一个通用“Agent Swarm 产品”。

没有必要暴露一大批产品化命名 Agent。runtime 的核心应始终围绕：

- `main.author`
- `fork.worker`
- `fresh.reviewer`

后续如有必要，可再增加：

- `fresh.researcher`

但这是后续增强，不是当前核心设计。

## 5. 页面生成模型

### 5.1 全局顺序

页面生成必须严格串行。

Catalog 完成后，顺序应为：

`第 1 页 -> review -> validate -> 第 2 页 -> review -> validate -> ...`

不允许跨页并行生成。

### 5.2 原因

Wiki 应当像一本书稿：

- 前文建立概念
- 后文承接前文
- 推荐阅读和交叉链接建立在已完成页面之上
- 整体结构会随着书稿推进而逐步稳定

### 5.3 允许的并行范围

并行只允许发生在单页内部。

例如：

- 多路证据搜索
- 多文件检查
- 多个小节的证据摘要
- 多个窄范围比较任务

主控 Agent 始终是唯一写作者和整合者。

## 6. 检索理念

### 6.1 硬决策

RepoRead 不使用：

- RAG
- embedding
- 向量检索
- SQLite
- 重型预构建代码索引

### 6.2 检索模型

RepoRead 使用：

`轻量清单 + 实时本地检索`

清单像图书馆目录卡片，只负责帮助导航和定位，不替代直接阅读源材料。

### 6.3 实时检索来源

主检索来源应是本地工具：

- `rg`
- `find`
- `git`
- 窗口化文件读取
- 已生成页面 Markdown

整个系统的体感应当更像一个认真读本地代码仓库的工程师，而不是一个语义召回产品。

---

## 7. 轻量清单策略

持久化只保留最轻量的文件型清单。

### 7.1 允许的持久化产物

- `wiki.json`
- `pages/<slug>.md`
- `pages/<slug>.meta.json`
- 页面级或版本级 citations 清单
- `version.json`
- `current.json`
- `job-state.json`
- review 摘要
- ndjson 日志

### 7.2 清单职责

这些清单允许承担：

- 导航
- 阅读顺序
- 页面元数据
- 章节关系
- 引用映射
- 恢复与续跑
- 版本浏览

它们不应偷偷演变成代码索引。

### 7.3 明确限制

不要加入：

- 符号数据库
- 代码 chunk 数据库
- 向量存储
- SQLite FTS 搜索层
- 大型生成式代码索引

系统应更多相信模型能力加实时本地检索，而不是额外的检索基础设施。

## 8. 工具体系

### 8.1 工具设计原则

RepoRead 应尽量复用成熟 coding agent 已经验证过的工具心智和命名。

除非 RepoRead 有很强的领域理由，否则不要重新发明一套平行术语。

### 8.2 推荐核心工具集

基线工具应来自 `claude-code`、`codex`、`opencode`、`oh-my-openagent`，再按 RepoRead 的只读边界做裁剪。

建议的高频工具：

- `Read`
- `Grep`
- `Glob` 或 `Find`
- `Bash`
- `Git`
- `Agent`
- `Task`
- `SendMessage`

RepoRead 自己的轻域工具可增加：

- `PageRead`
- `CitationOpen`

### 8.3 工具优先级

正常检索路径应为：

1. 当前页与页面清单
2. `Grep`
3. `Find` / `Glob`
4. `Read`
5. `Git`
6. `Bash` 作为受控兜底

`Bash` 不应是主抽象，只是本地只读操作的兜底壳。

### 8.4 工具复用原则

如果参考仓库里已经有成熟实现，并且符合 RepoRead 边界，就优先复制或改造，不重新设计。

RepoRead 应优先追求杠杆，而不是原创性。

---

## 9. 并行规则

### 9.1 并行原则

并行存在的目的是缩短单页内部证据收集时间，而不是把 RepoRead 做成一个分布式写书集群。

### 9.2 允许的并行

允许：

- 为单页内互不依赖的子任务并发拉起多个 `fork worker`
- 在一步内并行发起多个独立的读/搜任务

不允许：

- 同时写不同页面
- 同时由两个 Agent 写同一页
- 主控和已委派 worker 做完全重叠的探索

### 9.3 反重复原则

一旦主控已经把某个窄搜索或检查任务委派出去，除非该任务失败或证据不足，否则主控不应自己再完整重复同样的搜索。

这一点需要明确写进 Agent 行为规范。

## 10. 审稿模型

### 10.1 人类团队类比

RepoRead 的单页生产过程应更像一个技术书写作团队：

- `Author`：主控 Agent
- `Technical Reviewer`：fresh reviewer
- `Copy/Structure Checker`：确定性校验器
- `Editor/Publisher`：最终整理与发布

### 10.2 Reviewer 输入

fresh reviewer 应收到：

- 页面标题
- 章节位置
- 全书摘要
- 当前页草稿
- 当前页 citations
- covered files
- 明确的审稿问题

### 10.3 Reviewer 输出契约

reviewer 应返回紧凑结构化结果，例如：

- `verdict`
- `blockers`
- `factual_risks`
- `missing_evidence`
- `scope_violations`
- `suggested_revisions`

reviewer 不是替代作者，最终修稿责任仍在主控。

### 10.4 Reviewer 权限

reviewer 必须允许重新使用相同的本地只读检索工具独立复核。

这样才能降低：

- 上下文污染
- 主控过度自信
- 审稿流于形式

---

## 11. 模型配置

### 11.1 用户侧配置必须保持克制

不要给用户太大的配置面。

用户侧只暴露角色级模型配置：

- `main.author`
- `fork.worker`
- `fresh.reviewer`

每个角色只需要：

- `model`
- `fallback_models`

普通用户不需要再接触更多参数。

### 11.2 系统内部维护模型级 Prompt 调优

系统应内部维护按模型家族的 prompt tuning profile。

这层不对用户开放。

它可以内部调整：

- prompt append
- 输出框架
- review 严格度措辞
- tool-use 提示
- 冗长抑制
- 格式稳定性

这样既能适配不同模型差异，又不会把 RepoRead 变成一个复杂调参平台。

### 11.3 角色级降级链

fallback chain 必须是角色本地的。

例如：

- `fresh.reviewer` 失败时，应走 reviewer 自己的 fallback 链
- `fork.worker` 可以走更便宜或更快的 fallback 链
- 不能让 reviewer 静默继承 author 的降级策略

## 12. CLI 与 Web 体验

### 12.1 CLI 角色

CLI 是入口与过程面板：

- init/config/providers
- generate/jobs
- ask
- browse
- doctor
- versions

CLI 输出应反映“写一本书”的流程，例如：

- 正在写当前页
- 正在收集证据
- 正在审稿
- 正在修订
- 正在校验

而不是暴露一堆原始 Agent 内部术语。

### 12.2 Web 角色

Web 是长时阅读与追问界面：

- 书架/项目列表
- 版本浏览
- 左侧目录树
- 中间正文阅读
- 引用与源码抽屉
- 页面内 Chat Dock

搜索应先命中轻量清单，再按需回落到实时本地搜索。

### 12.3 用户心智

用户应把 RepoRead 感知成：

- 在读一本技术书
- 在需要时展开源码引用
- 在当前页面继续追问
- 在看到“审稿中 / 已修订 / 校验通过”这种贴近写作流程的状态

用户不需要理解 `fork worker` 和 `fresh reviewer` 这些内部术语。

---

## 13. 质量策略

### 13.1 不设硬性时间指标

RepoRead 不再对外承诺激进的墙钟时间目标。

文档里应移除分钟级生成承诺，改成：

- 质量优先
- 可恢复
- 可续跑

### 13.2 发布门槛

一页页面只有在满足以下条件后，才可视为可发布：

- review 通过，或 review 问题已修订完成
- 结构校验通过
- 引用校验通过
- 链接校验通过

### 13.3 恢复与续跑

生成过程允许更慢，这是可接受的。

因此系统必须更强调：

- 可恢复
- 可部分续跑
- 页面级进度持久化
- 清晰的重启点

---

## 14. 测试策略

### 14.1 测试重点

测试重点应放在：

- 串行页面写作行为
- 页内 fork 并行取证后的整合
- fresh reviewer 的隔离性
- 确定性校验门槛
- 中断后的恢复

### 14.2 必要测试层级

- 单元测试：工具封装、路径约束、清单落盘、状态恢复
- 集成测试：单页生成、fork worker fan-out/fan-in、reviewer 闭环
- Golden 测试：Catalog、单页输出、review 输出、CLI `ask` 输出格式
- E2E：`init -> generate -> interrupt -> resume -> browse -> ask`

### 14.3 人工验收重点

人工评审时应重点关注：

- 章节顺序是否自然
- 后文是否真正建立在前文之上
- reviewer 是否真的抓出了问题
- 检索体感是否足够本地、足够 grounded
- Web/CLI 是否更像一本技术书的阅读工作流，而不是一个通用 Agent 面板

## 15. 现有主文档需要的修改

### 15.1 `docs/prd.md`

需要重写为：

- 去掉硬性时间指标
- 去掉 RAG 叙事
- 改成“本地阅读/写书系统 + 独立审稿”的产品定位
- 明确页面按顺序串行生成
- 明确质量和可恢复性高于速度

### 15.2 `docs/design.md`

需要重写为：

- 去掉 SQLite 和重索引假设
- 去掉向量/语义召回叙事
- 改成“轻量清单 + 实时本地检索”
- 定义串行页面流水线和页内 fork 并行
- 明确以文件落盘作为主要持久化手段

### 15.3 `docs/agent-architecture.md`

需要重写为：

- 把大量命名 Agent 收敛成单主循环
- 明确只保留两种委派原语：`fork worker`、`fresh reviewer`
- 明确 review 是新会话、独立复核
- 明确角色级模型与 fallback 配置
- 明确系统内部维护模型级 prompt 调优，但不对用户开放
- 明确反重复探索原则和委派规则

---

## 16. 非目标

这次重设计不追求：

- 吞吐最大化
- 变成通用多 Agent 编排平台
- 暴露高级 prompt 调优给用户
- 依赖重型检索基础设施
- 为了新颖性放弃成熟复用

---

## 17. 验收标准

当且仅当以下条件成立时，这轮设计重构才算正确落进主文档：

1. 三份核心文档一致描述“一个主控循环”，而不是很多并列主 Agent
2. 所有 SQLite / RAG / 向量检索假设都被移除
3. 严格串行页面生成被明确写出
4. `fork worker` 与 `fresh reviewer` 的语义被明确写出
5. 角色级模型与 fallback 配置被明确写出
6. 模型级 prompt 调优存在于系统设计中，但不是用户可配置项
7. 工具体系以复用成熟 coding agent 命名和心智为主
8. 硬性时间指标被替换为质量优先表述
