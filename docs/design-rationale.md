## 1. 为什么要有这份文档

`prd.md`、`design.md`、`agent-architecture.md` 负责定义“做什么”和“怎么做”，但在长周期开发里，真正容易漂移的往往不是实现细节，而是设计判断本身。

这份文档专门回答：

1. 我们为什么收敛到当前这套方案。
2. 有哪些路线是明确讨论过并明确拒绝的。
3. 后续开发遇到模糊选择时，应该优先坚持什么。
4. 怎么判断实现已经开始偏离既定思路。

可以把它理解为 RepoRead 当前阶段的“设计决策备忘录”。

---

## 2. 总体判断

RepoRead 的目标不是做一个“功能更多、更像平台”的 Agent 系统，而是做一个真正适合人类长期使用的代码阅读与技术书写工作台。

这个判断包含 5 层含义：

1. 本地优先，而不是平台优先。
2. 质量优先，而不是速度优先。
3. 阅读与写作优先，而不是自治执行优先。
4. 人类体感优先，而不是技术炫技优先。
5. 主链路收敛优先，而不是角色膨胀优先。

因此，RepoRead 的设计核心不是“把 Agent 做得尽可能复杂”，而是“把代码阅读、页面生成、问答、研究这条主链路做得稳定、清晰、可信”。

---

## 3. 我们最终确认的产品心智

RepoRead 应被理解为：

- 一个本地优先的仓库阅读工作台。
- 一个围绕页面组织知识、围绕页面追问问题、围绕页面沉淀研究结果的系统。
- 一个由同一套主控 Agent 统一驱动的阅读与写作环境。

它不是：

- 一个通用 RAG 平台。
- 一个 Agent Swarm 控制台。
- 一个用户可以自由改 prompt、随意编排角色的平台。
- 一个自动改代码、自动提 PR 的执行代理。

一句话总结：

> RepoRead 是“会读代码、会写页面、会基于页面继续问和继续研究”的本地工作台，不是“把一切 Agent 能力都堆上去的平台”。

---

## 4. 这轮设计最重要的取舍

### 4.1 质量优先，不设激进时间目标

我们明确不把“生成很快”当成第一目标，也不鼓励为了表面吞吐去牺牲页面质量。

原因：

1. 这类产品的核心价值是长期可读、可追问、可复用，而不是一次性秒出摘要。
2. 页面一旦成为后续问答和研究的基础，质量不足会被后续链路放大。
3. 用户真正能感知的不是单次快 10 秒，而是结果是否可靠、结构是否稳定、审稿是否独立。

开发含义：

- 不用把时间指标写得过于激进。
- 优先保证证据完整、结构稳定、恢复清晰。
- 允许耗时更长，只要质量收益明确。

### 4.2 不走 RAG / 向量 / SQLite 路线

我们明确反对把 RepoRead 设计成数据库化、索引化、向量化的知识系统。

原因：

1. 用户希望的是更接近人类阅读习惯的本地检索体验，像 `grep`、`find`、`git`、窗口化读文件，而不是黑箱语义召回。
2. 重型检索底座会显著抬高系统复杂度、调试成本和心智负担。
3. 这类产品的可信感很大程度来自“我知道你是怎么找到这段内容的”，而不是“系统告诉我它召回到了相关片段”。
4. 当前目标是逆向并超越 Zread，但超越点应主要来自 Agent loop、页面质量、问答能力、交互体验，而不是换一套重型基础设施。

开发含义：

- 不引入向量库、embedding、SQLite/FTS、常驻索引服务。
- 存储以 Markdown + JSON + 轻量清单为主。
- 检索以 `PageRead -> Grep -> Find -> Read -> Git -> Bash` 为主路径。

### 4.3 Agent 架构必须收敛，不能平台化膨胀

我们明确认为早期版本的 Agent 设计不应该过于复杂。

原因：

1. 强如 `oh-my-openagent` 也不靠堆很多命名子 Agent 获得核心价值。
2. 角色越多，状态同步、上下文治理、失败恢复和调试难度越高。
3. 文档生成和代码问答本质上都需要一个清晰的主控心智，而不是一堆并列 controller。

最终收敛结论是：

- 只有一个主控实现：`main.author`
- 只有两种委派原语：`fork.worker`、`fresh.reviewer`
- 只有一个确定性校验器：`validator`

也就是说：

> 单主循环，不做多主控；保留必要委派，不做角色军团。

### 4.4 页面必须严格串行，不能按章节并发写

我们明确不建议把章节或页面级生成做成并发写作。

原因：

1. 页面之间存在阅读顺序和概念依赖。
2. 并发写作会放大术语漂移、内容重叠、边界冲突和交叉引用错误。
3. 代码阅读型文档不是 FAQ 拼接，章节间是有叙事连续性的。

但我们同样确认：

- 单页内部的检索、局部摘要、信息汇总可以并行。

最终原则：

- 跨页严格串行。
- 页内允许并行取证和局部归纳。
- 最终成文只能由单一主控收束。

### 4.5 Review 必须独立成角色，而且必须是新会话

我们明确保留单独的审稿角色，而且强调必须新会话拉起。

原因：

1. 主控作者极易出现“上下文污染”和“自信偏差”。
2. 同一个会话里让模型又写又审，审稿会退化成自我合理化。
3. 类似人类图书初版团队，写作者和审稿者应承担不同职责。

最终原则：

- `fresh.reviewer` 必须是 fresh session。
- 审稿 briefing 必须完整，不依赖继承作者上下文。
- reviewer 只负责指出问题、风险、遗漏和结构缺陷，不直接接管写稿。

### 4.6 子 Agent 拉起方式要向 Claude Code 学习

我们明确希望委派方式更像 Claude Code，而不是随意开分身。

核心要求：

1. 每次拉起都要是新的清晰任务。
2. 任务说明必须完整，尤其是 `fresh.reviewer`。
3. `fork.worker` 只做窄范围局部任务，不允许上下文发散。
4. 子任务的返回结果必须结构化，便于主控整合。

也就是：

- `fork.worker` 类似 Claude Code 的上下文继承型 worker。
- `fresh.reviewer` 类似 Claude Code 的 fresh subagent，用于 second opinion / 独立审稿。

### 4.7 模型配置要有角色隔离，但不向用户暴露过多灵活性

我们明确认可：

- 不同角色支持不同模型配置。
- 支持多级降级。
- 支持面向不同模型族的系统内建 prompt 调优。

但同时明确反对：

- 把这些内部能力大面积暴露成用户级开关。

原因：

1. 用户通常并不想维护一套 prompt 工程平台。
2. 过度灵活会迅速转化为可用性下降和调试负担。
3. 模型适配应由系统维护，而不是转嫁给用户。

最终原则：

- 用户只配置少量角色级模型映射。
- 系统内部维护 prompt tuning profile、fallback chain、能力探测和降级策略。
- 不提供用户级 prompt override。

### 4.8 工具设计要尽量复用成熟实现

我们已经明确：

- 不要为了“原创”而重造轮子。
- 只要四个参考仓库里有现成可用的优秀设计，都应该优先吸收。

参考方向：

| 仓库 | 重点吸收点 |
| --- | --- |
| Claude Code | 单主循环、委派语义、上下文压缩、工具边界 |
| Codex | 本地优先、清晰 runtime、严格工具调用约束 |
| OpenCode | Web/CLI 一致性、Provider 无关设计、终端体验 |
| oh-my-openagent | 角色级模型配置、降级链、模型族调优思路 |
| Zread | Catalog/Page 主链路、版本化 wiki、本地预览闭环 |
| DeepWiki-Open | 页面化知识组织、问答与研究视角 |

原则不是“参考一下”，而是：

> 能直接复用的设计和实现，就直接复用；差异化应主要来自更好的整合，而不是为了避嫌而另起一套。

## 5. 后续开发必须坚持的结构性原则

### 5.1 一个主控，四种 mode

主控只有一个：`main.author`。

所有工作都在四种 mode 中切换：

- `catalog`
- `page`
- `ask`
- `research`

如果后续实现开始出现“为每种 mode 再造一个并列主 Agent”的趋势，基本就已经偏航。

### 5.2 检索优先于生成，证据优先于表达

先把证据找全，再决定怎么写。

顺序要稳定：

1. 先读已有页面和引用。
2. 再做本地实时检索。
3. 再深入读取源码。
4. 最后再组织写作。

不能反过来先写一个大概，再回头补证据。

### 5.3 文件系统是真实状态源

`.reporead` 目录和其中的 Markdown / JSON / 清单文件是系统的真实状态源。

这意味着：

- 任务恢复依赖文件状态。
- 版本切换依赖文件状态。
- 页面、审稿、校验、研究产物都应落盘可查。

如果后面开始把关键状态大量转移到内存对象或隐式缓存里，也是在偏航。

### 5.4 Web 和 CLI 只是两个壳，不是两套系统

`packages/core` 才是唯一运行时。

Web 和 CLI 都只是消费同一个核心服务、同一套事件流、同一套落盘协议。

不能出现：

- CLI 一套生成逻辑，Web 另一套生成逻辑。
- CLI 有恢复语义，Web 没有。
- Web 能问答，CLI 只能生成。

## 6. 明确不要做的方向

以下方向都讨论过，并且当前阶段明确不做。

### 6.1 不做“重型知识底座”

- 不做向量库。
- 不做 embedding 管线。
- 不做 SQLite / FTS。
- 不做预构建重型全文索引。
- 不做常驻后台索引服务。

### 6.2 不做“复杂可见 Agent 平台”

- 不做 Agent 市场。
- 不做多主控编排台。
- 不做用户可视化 agent workflow 编辑器。
- 不做用户侧的复杂 prompt chaining。

### 6.3 不做“过度灵活的用户配置”

- 不做用户级 prompt 覆盖。
- 不做大量高级 provider 调参面板。
- 不做让用户理解内部委派拓扑的配置项。

### 6.4 不做“吞吐优先”的文档生成器

- 不用跨页并发写作换速度。
- 不为了缩短生成时间降低审稿或校验强度。
- 不把快当作默认卖点。

---

## 7. 开发时的判断优先级

当实现过程中出现多个看起来都能成立的方案，按下面顺序判断：

1. 是否更符合人类阅读和使用习惯。
2. 是否更有利于页面质量和证据可信度。
3. 是否更有利于主链路收敛，而不是角色膨胀。
4. 是否更容易恢复、调试和解释。
5. 是否能复用成熟设计和现成实现。
6. 只有在以上都不冲突时，才考虑是否更快。

这条优先级非常重要。

RepoRead 的优化方向是：

> 先保证“像人类在认真读代码和写技术内容”，再考虑“像一个高吞吐系统”。

---

## 8. 开发偏航检查表

如果后续开发中出现下面任一现象，需要主动回看本文件：

1. 方案开始引入向量库、embedding、SQLite、全文检索数据库。
2. 页面开始按章节或多页并行生成。
3. 审稿者和作者共享同一上下文或同一会话。
4. 新增了很多命名 Agent，每个都想拥有自己的主流程。
5. 工具链开始偏离 `grep/find/git/read` 这种可解释的本地检索方式。
6. 用户需要理解越来越多内部概念才能完成基本配置。
7. Web 和 CLI 的主功能开始不一致。
8. 模型适配逻辑开始要求用户手动调 prompt 才能用。
9. 实现为了“更快”牺牲了页面证据、审稿独立性或恢复语义。
10. 关键状态越来越多地存在内存中，而不是可审阅的 Markdown/JSON 文件里。

## 9. 后续实现时应如何使用这份文档

建议把这份文档当成 3 类场景下的判断基准：

### 9.1 做架构决策时

先问自己：

- 这个设计是在增强主链路，还是在扩张平台形态？
- 它更像“本地阅读工作台”，还是更像“研究平台/Agent 平台”？

### 9.2 做功能评审时

先问自己：

- 它有没有破坏严格串行、独立审稿、轻量清单、本地检索这几条底线？
- 它有没有把系统复杂度转嫁给用户？

### 9.3 看代码是否跑偏时

先问自己：

- 这段实现是否还能让人一眼理解 RepoRead 的核心心智？
- 还是已经开始长成另一类产品？

---

## 10. 最后的约束性总结

如果要把这轮讨论压缩成几句话，应该是：

1. RepoRead 首先是本地优先的代码阅读与技术书写工作台。
2. 生成、问答、研究必须共享同一套页面化知识结构。
3. Agent 架构要像 Claude Code 那样收敛：单主循环，有限委派，严格边界。
4. 页面生成必须严格串行，审稿必须 fresh session 独立执行。
5. 检索要像人类读代码一样，以本地实时检索和文件阅读为主，而不是走 RAG/向量/数据库路线。
6. 模型适配、prompt 调优和降级策略由系统维护，不把复杂性暴露给用户。
7. 可以大量复用成熟开源项目的优秀设计与实现，不需要为“原创”而牺牲质量和效率。

这 7 条如果后续还能一直成立，RepoRead 的方向大概率就是稳定的。
